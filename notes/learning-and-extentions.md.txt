{"textColorIndex":0,"bgColorIndex":0,"note":"# Learnings & Extensions\n\n## What this project covered\n\nBuilt an end-to-end recommender system data pipeline in three stages: a batch ETL pipeline preparing training data from transactional ratings, a PostgreSQL + pgvector database storing model-generated embeddings, and a streaming pipeline using Kinesis and Lambda to serve real-time product recommendations from live user activity.\n\n## Key takeaways\n\n**Vector databases solve a specific problem regular tables don't.** Storing embeddings as plain columns in a relational table works, but finding the \"most similar\" vectors to a given query at scale requires specialized indexing for high-dimensional similarity search. pgvector's distance operators and indexing structures (HNSW, IVFFlat) make this efficient. The deeper insight: this exact pattern — embed, store, retrieve-by-similarity — is the same architecture behind retrieval-augmented generation (RAG) in LLM applications. The vectors here represent products and users; in a RAG system they'd represent text chunks. The infrastructure problem is identical.\n\n**Batch and streaming pipelines often share a data foundation but serve different latency needs.** The batch pipeline prepares training data on a schedule (or on-demand), tolerating latency in exchange for processing larger volumes efficiently. The streaming pipeline trades that efficiency for low latency, processing events as they arrive so recommendations stay current with live user behavior. Designing both around the same underlying model artifacts (not duplicating logic) is what makes the architecture coherent.\n\n**Partitioning shows up at every stage for the same reason.** The training data was partitioned by `customerNumber`, and the recommendations output was partitioned by `year\/month\/day\/hour`. Different partition keys, same underlying goal: let downstream consumers (a data scientist querying training data, or a monitoring process checking recent recommendations) scan only the data they need instead of the entire dataset.\n\n**Infrastructure credentials need careful handling even in a lab.** Configuring the Lambda's database connection meant handling a real username\/password pulled from Terraform outputs. Treating those values as sensitive by default — never screenshotting them, never hardcoding them into code — is a habit that should carry over directly into production work, not just something to remember \"when it matters.\"\n\n## What I'd do differently \/ extend\n\n- Pull and document the actual vector similarity query inside the inference Lambda — I worked through the conceptual mechanism (likely a `<->` or `<=>` nearest-neighbor query against the `item_emb` table) but didn't get to inspect the literal code before the lab session expired. This is the first thing to revisit if I get lab access again.\n- Add monitoring (CloudWatch Alarms + SNS) on both Lambda functions and the Firehose delivery stream, rather than relying on manually checking the Monitor tab\n- Move the Lambda environment variables (currently set manually through the console) into Terraform so the entire pipeline, including runtime configuration, is reproducible from code\n- Test the pipeline under a more realistic, bursty event pattern rather than the lab's steady ~10-second interval, to see how Firehose buffering and Lambda concurrency behave under load\n- Explore extending the same vector retrieval infrastructure into a small RAG prototype — same pgvector backbone, different embedding source (text instead of user\/item vectors)\n\n## Connection to AI engineering\n\nThis project sits at the intersection of data engineering and AI infrastructure. The batch and streaming pipeline design is standard data engineering practice, but the vector database stage is foundational AI infrastructure — the same pattern (embed → store → similarity search → retrieve) is what makes modern LLM applications like RAG systems and semantic search possible. Understanding this pipeline end-to-end is directly transferable to building retrieval infrastructure for LLM-based applications, not just recommender systems."}
